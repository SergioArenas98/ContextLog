## Feature module layout (consistent across all 7 modules)

```
lib/features/<entity>/
├── data/
│   ├── tables/
│   │   └── <entity>_table.dart    ← Drift Table definition
│   └── repositories/
│       └── <entity>_repository.dart ← CRUD methods, row→model mapping
├── domain/
│   ├── models/
│   │   ├── <entity>_model.dart    ← @freezed model definition
│   │   └── <entity>_model.freezed.dart ← Generated (do not edit)
│   └── validators/                ← (present for most entities)
│       └── <entity>_validator.dart
└── presentation/
    ├── providers/
    │   └── <entity>_providers.dart ← Repository provider, list providers
    └── widgets/  (or screens/)
        ├── <entity>_list_tab.dart  ← List view inside feature detail
        └── <entity>_form_sheet.dart ← Create/edit bottom sheet
```

**Exception**: the `feature` module has `screens/` instead of `widgets/` for the full-page views:
- `feature_list_screen.dart`
- `feature_detail_screen.dart`
- `feature_form_screen.dart`
- `context_station_panel.dart` — overlay panel shown in the feature detail screen
- `widgets/feature_card.dart`

The `context` module also has a `screens/` subdirectory:
- `context_focus_screen.dart` — full-page context detail/edit view (`/features/:id/contexts/:cid`)

The `harris_matrix` module has a `screens/` subdirectory:
- `matrix_full_screen.dart` — full-screen interactive Harris Matrix (`/features/:id/matrix`)

## Naming conventions

| Pattern | Example |
|---|---|
| Feature module | `lib/features/find/` |
| Table class | `FindsTable` |
| Repository class | `FindRepository` |
| Model class | `FindModel` |
| Validator class | `FindValidator` |
| Provider file | `find_providers.dart` |
| List provider | `findsByFeatureProvider` |
| Repository provider | `findRepositoryProvider` |
| Validator provider | `findValidatorProvider` |
| Screen/tab widget | `FindListTab`, `FindFormSheet` |

## Important notes

- `*.freezed.dart` and `app_database.g.dart` are **generated files** — do not edit manually; regenerate with `dart run build_runner build --delete-conflicting-outputs`
- All feature modules follow the same three-layer structure; deviating will break the pattern
- `core/widgets/` contains only shared, reusable widgets; feature-specific widgets stay in their module
- The `harris_matrix` feature's `presentation/widgets/` contains the matrix painter and the relation form — there is no separate `harris_matrix_list_tab` because the matrix is a canvas, not a list
